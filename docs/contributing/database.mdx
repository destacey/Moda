# Database & Migrations

## Database Architecture

Wayd uses a **single shared database** accessed through `WaydDbContext`. All domain services share this context.

Entity configurations are organized by domain area in:

```
Wayd.Infrastructure/src/Wayd.Infrastructure/Persistence/Configuration/
```

## Entity Framework Core

### Direct DbContext Usage

Wayd does **not** use the Repository pattern. Application handlers access `WaydDbContext` directly:

```csharp
public class GetWorkItemQueryHandler : IQueryHandler<GetWorkItemQuery, WorkItemDto>
{
    private readonly WaydDbContext _context;

    public async Task<Result<WorkItemDto>> Handle(GetWorkItemQuery request, CancellationToken ct)
    {
        var workItem = await _context.WorkItems
            .Where(w => w.Id == request.Id)
            .ProjectToType<WorkItemDto>()
            .FirstOrDefaultAsync(ct);

        return workItem is null
            ? Result.Failure<WorkItemDto>("Work item not found.")
            : Result.Success(workItem);
    }
}
```

### Entity Configuration

Use fluent API in separate configuration classes:

```csharp
public class WorkItemConfiguration : IEntityTypeConfiguration<WorkItem>
{
    public void Configure(EntityTypeBuilder<WorkItem> builder)
    {
        builder.ToTable("WorkItems", SchemaNames.Work);
        builder.HasKey(w => w.Id);
        // ...
    }
}
```

## Soft Deletes

Wayd uses soft deletes throughout ŌĆö records are never permanently removed from the database. Instead, they are marked as deleted via the `ISoftDelete` interface.

### BaseSoftDeletableEntity

Entities that support soft delete inherit from `BaseSoftDeletableEntity`, which implements `ISoftDelete`:

```csharp
public abstract class BaseSoftDeletableEntity<TId> : BaseAuditableEntity<TId>, ISoftDelete
{
    public Instant? Deleted { get; set; }
    public string? DeletedBy { get; set; }
    public bool IsDeleted { get; set; } = false;
}
```

When `SaveChangesAsync` is called, `BaseDbContext` intercepts any `EntityState.Deleted` entry that implements `ISoftDelete`, sets `IsDeleted = true`, and changes the state back to `Modified` ŌĆö so nothing is ever physically deleted.

### Global Query Filter

`BaseDbContext.OnModelCreating` registers a global EF query filter for all `ISoftDelete` entities:

```csharp
modelBuilder.AppendGlobalQueryFilter<ISoftDelete>(s => !s.IsDeleted);
```

This means all LINQ queries automatically exclude soft-deleted rows ŌĆö no manual `Where(x => !x.IsDeleted)` is needed in handlers or validators. Use `.IgnoreQueryFilters()` only when you explicitly need to include deleted records.

### Unique Indexes on Soft-Deletable Entities

Because soft-deleted rows remain in the table, a plain unique index will block reuse of a deleted value. Always add `HasFilter("[IsDeleted] = 0")` to any unique index on a soft-deletable entity:

```csharp
builder.HasIndex(r => r.Name).IsUnique().HasFilter("[IsDeleted] = 0");
```

This keeps the constraint consistent with what the global query filter enforces at the application layer.

## Migrations

Migrations are managed in `Wayd.Infrastructure.Migrators.MSSQL`.

### Creating a Migration

```bash
dotnet ef migrations add <MigrationName> \
  --project "Wayd.Infrastructure/src/Wayd.Infrastructure.Migrators.MSSQL" \
  --startup-project "Wayd.Web/src/Wayd.Web.Api"
```

### Applying Migrations

```bash
dotnet ef database update \
  --project "Wayd.Infrastructure/src/Wayd.Infrastructure.Migrators.MSSQL" \
  --startup-project "Wayd.Web/src/Wayd.Web.Api"
```

Migrations also run automatically on application startup via `app.Services.InitializeDatabases()`.

## Database Initialization

On startup, the application:

1. Applies pending migrations
2. Seeds feature flags from code definitions
3. Runs any data initialization logic
